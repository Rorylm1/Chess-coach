# Chess Coach — Project Spec & Milestones

## Context

A **personal web-based chess coach** that makes studying chess enjoyable, consistent, and genuinely useful — sitting between raw engine analysis and human learning. The core promise: after every meaningful chess moment, answer *"What is the useful lesson here?"* in plain language, grounded in engine truth, delivered by a warm playful-mentor coach.

This spec builds on three source docs in this folder:
- `objective.md` — product vision, principles, V1 focus, opening-learning vision, coach personality.
- `deep-research-chess-coach.md` — architecture (Next.js/Vercel, react-chessboard + chess.js, browser Stockfish WASM, server LLM coaching), licensing analysis, dependency shortlist, risks.
- `distinctive_design.md` — anti-"AI-slop" frontend strategy (frontend-design skill, design tokens, banned-defaults, Motion for React).

Five product decisions shape the milestone structure:

1. **Milestones are outcome-based** — defined by what works end-to-end, each independently demoable, no time-boxes.
2. **Coaching LLM = Claude (Anthropic)** — latest Claude model via the Anthropic API, behind a thin provider abstraction.
3. **Opening learning is the priority feature.** The research doc deferred it; we elevate it into the V1 arc as the centerpiece (M5), placed *after* the grounding spine and coaching layer it depends on.
4. **Full design investment from day one.** Distinctive design is first-class in every milestone, anchored by a design system established in M0.
5. **Post-M5 pivot — Play becomes its own destination, and chess gets fun + social.** With the grounding spine (M2), coach (M3), and opening journeys (M5) all shipped, the app is reorganized into **three tabs — Play · Openings · Coach** (a 4th may follow). The focus shifts to making **Play** a place you *want* to be: a per-game **randomized board** (every game looks one-of-a-kind) and **playing a real human**. The new milestone arc: **M6** = Play+Openings tabs + the randomized board (built first) + local hot-seat · **M7** = a **shareable live link** for two people to play online · **M8** = the **Coach** tab (post-game review home) · **M9** = release readiness. The original "remember my recurring weaknesses" arc (the old M6) is **deferred** to `ideas.md` — the personal-coach long-game still stands, it's just no longer the next thing we build.

## Architecture (decided)

The non-negotiable principle: **separation of calculation from teaching.** The engine is the source of truth for chess facts; Claude is constrained to *narrate* those facts, never to invent them.

Three cleanly separated layers:

- **Play & board (client):** `chess.js` (BSD-2-Clause) for rules/legal moves/PGN/FEN, with a **custom hand-built board** for UI (M1 shipped this in place of the originally-planned `react-chessboard` — see the M1 "✅ Shipped" note for why). All move validation local and instant.
- **Analysis (client, Web Worker):** Stockfish compiled to WASM (`nmrugg/stockfish.js`, GPLv3), off the main thread. Produces eval (cp/mate), best move, principal variation. **Start with the single-threaded "lite" build** to avoid COOP/COEP cross-origin-isolation complexity on Vercel; upgrade later if needed.
- **Coaching (server, Vercel Function):** Next.js route handler receives a *structured engine payload* (FEN, side to move, move played, eval before/after, eval delta, best move(s)+PV in SAN, deterministic classification) and calls **Claude** to explain *why it matters*. API key in a server-side env var.

**Stack:** Next.js (App Router) on Vercel; GitHub as source of truth (prod from `main`, preview deploy per PR). Local-first storage via **IndexedDB** (`idb`). Deterministic move classification (blunder/mistake/inaccuracy) computed **in code** from eval-delta / win-probability swings — never by the LLM.

**App shape (post-M5):** three top-level tabs in the long-term vision — **Play** (vs the bot or a human; randomized board; coach available in **bot mode only**), **Openings** (the M5 journeys), **Coach** (the grounded teaching / post-game review surface). They land across milestones: M6 ships **Play + Openings**, M8 adds **Coach**. Within Play, a `Bot · Multiplayer` mode switch drives one generalized game loop. Local-first stays the default; **online play (M7) is the one deliberate exception** — two players sharing live game state needs a small realtime relay (a hosted realtime service or P2P + a thin signaling endpoint). That relay carries *only* the live game (moves + the board seed); it is a transport, not a database of record, and it never touches the Stockfish/UCI boundary or our permissive-code guardrail (a service, not a GPL JS lib). Hot-seat play (M6) needs no backend at all.

**Licensing guardrail (carry through every milestone):** keep *our* code on a permissive foundation. Treat Stockfish WASM as a deliberately isolated component (static asset in `public/`, communicated with only via the UCI text protocol over Worker messages) so our app code is never a derivative work. **Avoid GPL JS libraries** (`chessground`, `chessops`) entirely. Be prepared to publish the engine's corresponding source + a GPLv3 §6 notice (M7).

## Design Approach (first-class throughout)

Design is invested in from day one. M0 establishes the system; every milestone ships UI that meets it.

- Install Anthropic's **frontend-design** skill so it auto-activates on UI work.
- Author **`DESIGN.md`** (loaded every session): semantic CSS variable tokens, a *named aesthetic direction*, and an explicit **banned-defaults** list (no Inter/Roboto/system fonts, no purple-indigo gradients, no glassmorphism, no three-identical-card grids, no gradient text on metrics, no uniform border-radius, no bounce easing).
- **Aesthetic direction (proposed, finalize via short exploration):** a *warm editorial study* feel — the coach as a knowledgeable, encouraging mentor, not a cold engine dashboard. Distinctive display + refined body type pairing; a committed dominant color with one or two sharp accents; intentional spatial rhythm; restraint plus one or two signature motion moments. Explicitly **not** neo-brutalist, not generic SaaS, not "engine UI with a friendly label."
- **Motion:** Motion for React (default); GSAP only for genuine timeline/scroll storytelling (e.g. an opening-journey walkthrough). Always ship `prefers-reduced-motion` fallbacks.
- **Workflow:** pull reference screens (Mobbin/Refero for patterns, Godly for motion) into `/screenshots`; do throwaway mockups in `research/designs/` before production code; have the agent screenshot its own output and self-correct.
- Bake WCAG basics (focus states, contrast, alt text) into `DESIGN.md`.

Strength uses `Skill Level` (0–20) + per-move think time; `UCI_Elo` was not used (its ~1320 floor can't express genuine-beginner play).

### M2 — Engine analysis & the grounding spine
The shared contract coaching *and* openings depend on. Build once, cleanly.
- Engine eval/PV plumbing: eval (cp/mate), best move(s), PV in SAN at a given depth; shallow for in-game, deeper for review.
- **Deterministic move classification** (blunder/mistake/inaccuracy) in code, from eval-delta / win-probability swings.
- The **structured engine-fact payload** type: FEN, side to move, move played (SAN), eval before/after, delta, best move(s)+PV (SAN), classification. The only thing coaching may reason from.
- **Move-validation utility:** validate any move string against `chess.js` legal moves + engine PV (used to reject/regenerate hallucinated LLM moves).
- Eval graph + color-coded move list UI.
- **Done when:** eval, classification, and PV render correctly for any position; the payload type is the single source of chess facts; the validator rejects illegal/non-PV moves.
- **Key files:** `lib/engine/analysis.ts`, `lib/classify.ts`, `lib/grounding/payload.ts`, `lib/grounding/validate-move.ts`, `components/EvalGraph/`, `components/MoveList/`.
- **✅ Shipped (M2).** The grounding spine is live in-game on `/play`. Actual files (under `src/`): pure eval vocabulary in `lib/engine/analysis.ts` (the `Eval` type normalized to White's POV, win-probability math, `formatEval`, `terminalEval`, in-game/review depth budgets) — the *engine search* itself stays in `lib/engine/engine.ts` as `ChessEngine.analyse()` so the Worker lives in exactly one place; `lib/classify.ts` (win-probability classification + quality metadata); `lib/grounding/payload.ts` (the `MoveFact` payload type + `buildMoveFacts`, the single source of chess facts); `lib/grounding/validate-move.ts` (the anti-hallucination validator: SAN/UCI, annotation-tolerant, with PV/best-move membership). UI: `components/EvalGraph/` (eval readout + best line in SAN + win% swing graph), `components/MoveList/` (classification-coloured log), plus an added `components/EvalBar/` (vertical bar beside the board) and the live `components/Play/useAnalysis.ts` hook. Decisions: (1) a **dedicated second full-strength engine** runs analysis (separate from the deliberately-weakened bot) so its judgement is honest and runs in parallel with the bot's thinking; (2) classification is **win-probability based, not raw centipawns**, so a +12→+6 move in a won position isn't a false "blunder" (the documented chess.com pain point); (3) evals are **cached by FEN** (takebacks/new games/transpositions self-heal, no reset); in-game search is depth 14, review depth 18 is defined for M4; (4) analysis is **on-demand** — a toggle (default on); the eval bar hides ≤640px so the board keeps full width; (5) added a **race-safe engine abort** (consume the post-`stop` `bestmove` before advancing the queue) so a stale search can never be misattributed. Verification: **vitest** added with 28 unit tests covering `classify` (blunder/mistake/inaccuracy tiers, the win-prob-beats-cp case, mover-perspective symmetry) and `validate-move` (illegal/gibberish rejected, SAN+UCI+promotion+castling accepted, PV membership) — run with `npm test`.

### M3 — The Coach (Claude coaching layer)
Turn engine facts into teaching — the make-or-break quality layer.
- Server route → **Claude**, behind a thin provider abstraction. API key in Vercel env var.
- Strict engine-grounded prompt: pass *only* the M2 payload; forbid inventing evals/ratings; bridge notation to plain language ("Qxh7+ — queen takes h7 with check"); ground in coordinates ("the knight on f3").
- **Anti-hallucination loop:** validate every move Claude names against `chess.js` + PV (M2 validator); reject/regenerate on miss.
- Streaming responses; single warm playful-mentor persona (celebrates good ideas, names mistakes gently, surfaces *one* useful lesson).
- **On-demand tiered hints** during play (nudge → concept → candidate move → full line), non-intrusive, never auto-revealing, shallow-depth engine output.
- Per-position explanation caching to control LLM cost.
- **Done when:** Rory can ask for a hint mid-game and get a grounded, tiered nudge; a finished game produces specific, correct key-moment explanations that never cite an illegal move.
- **Key files:** `app/api/coach/route.ts`, `lib/coach/prompt.ts`, `lib/coach/provider.ts`, `lib/coach/cache.ts`, `components/HintButton/`, `components/Coaching/`.
- **✅ Shipped (M3).** The coach is live in-game on `/play`. Actual files (under `src/`): `app/api/coach/route.ts` (the `POST /api/coach` route — `nodejs` runtime — that resolves engine facts into one human lesson and streams it back word-chunked); `lib/coach/payload.ts` (the `CoachPayload` the route accepts — a coaching-facing projection of the M2 `MoveFact`); `lib/coach/prompt.ts` (the strict engine-grounded system/user prompt + warm playful-mentor persona); `lib/coach/provider.ts` (the thin Anthropic abstraction: `isCoachConfigured`, `generateCoaching`, and a deterministic engine-only `coachFallback` for when the key is absent or the call fails); `lib/coach/guard.ts` (`hasUngroundedMove` — scans the generated prose for any move it named and rejects anything outside the allowed/legal/PV set); `components/HintButton/HintButton.tsx` + `components/Coaching/CoachPanel.tsx`. Decisions vs the plan: (1) **grounding is structural first, validated second** — the prompt only lets Claude quote a *closed move vocabulary*, and the route is the backstop: generate → scan → validate each named move against `chess.js` + the supplied PV → **regenerate once** on a miss → fall back to the deterministic engine-only explanation; the UI therefore can never display a move the engine didn't offer. (2) First ship **streams the already-validated text** (word-chunked for a live feel) rather than raw token-streaming, so validation happens *before* anything reaches the screen; the text/plain delta contract is forward-compatible with true token-streaming later. (3) `lib/coach/cache.ts` (per-position caching) is **deferred** — the validate-before-show loop and on-demand calls keep cost bounded for a single user; caching lands when usage warrants it. Verification: **vitest** covers `guard` (ungrounded/illegal move detection) and the coach `payload` shape (part of the now-81-test suite); browser-verified tiered hints + finished-game key-moment explanations with no illegal move ever cited.

### M5 — Opening learning (PRIORITY FEATURE)
The centerpiece. Openings as *ideas*, not move memorization. Depends on M1 + M2 (reuses the M3 coach voice for baked prose). _Design locked via grilling — see decisions below._
- **Scope:** guided **journeys** for **4 curated openings** — Italian Game & Queen's Gambit (you play White), Sicilian Defense & French Defense (you play Black). Chosen for canonical fame, teachability, and structural variety (open / closed / defense, across 1.e4 and 1.d4). The content format is uniform so growing toward ~20 openings later is data entry, not a redesign. _In-game opening awareness was split out → `ideas.md`._
- **Content model — the grounding extension:** opening *ideas* (plans, pawn structures, piece placement, traps) aren't engine-computable, so they live in a researched, **human-reviewed** knowledge layer in `content/openings/` (typed TS data, one file per opening). The spine still holds: the **engine verifies every concrete line + eval**, prose stays tied to sourced theory, and **all 4 are read/approved once before shipping**. The Lichess opening explorer is queried **at build time only** to source realistic continuations, then baked in — **no runtime Lichess calls** (rate limits never bite). Runtime journeys are fully static → instant, ~$0 LLM, zero hallucination risk. (Deferred nicety: a live "ask the coach a follow-up" box.)
- **A journey = two phases on one `/openings/[slug]` page:**
  - **Read-through** — the board steps down the main line, one coach note per move (existing playful-mentor voice), with **deviation asides** wherever the bot may branch (a main-line spine; branches demoted to footnotes), then thematic panels: *what each side wants · where the pieces belong · the signature trap · the middlegame it becomes*. The M2 `EvalBar`/`EvalGraph` show the engine eval per step (grounding).
  - **Recall drill** — you play your side (~8 plies) against a **scripted "book bot"** (no engine; picks among the curated taught replies, weighted by explorer popularity) that deviates *only into taught lines*. A non-book move → **gentle correct-and-retry** with a reveal escape hatch.
- **Narrow & taught:** the per-opening **book tree** (main line + 1–2 popular taught deviations per turn, ~8 plies) is the single source for *both* phases — you can only be drilled on what the read-through taught.
- **Personalization hook (deferred seam):** leave a hidden "your mistakes in this opening" slot; no work in M5. No progress persistence in M5 — persistence + weakness tracking is the deferred "Remember" arc (`ideas.md`), no longer a near-term milestone.
- **Done when:** Rory can pick one of the 4 openings, walk a guided board-illustrated read-through of its ideas, then pass a recall drill where the book bot varies its replies within the taught lines and the coach corrects off-book moves.
- **Key files:** `app/openings/page.tsx`, `app/openings/[slug]/page.tsx`, `content/openings/*.ts`, `lib/openings/tree.ts` (book-tree types + drill logic), `lib/openings/build/` (build-time explorer fetch + engine verification), `components/OpeningJourney/`; reuses `components/Board/`, `components/EvalBar/`, `components/EvalGraph/`, and the coach voice from `lib/coach/`.
- **✅ Shipped (M5).** Four journeys live under `/openings`: **Italian Game** & **Queen's Gambit** (play White), **Sicilian Defense** & **French Defense** (play Black) — each statically prerendered (`/openings/[slug]`). Actual files (under `src/`): `lib/openings/tree.ts` (the `Opening`/`BookMove` model + `readSteps`, `pickWeighted`, `matchLearnerMove` — the single source for *both* phases); `content/openings/{italian-game,queens-gambit,sicilian-defense,french-defense}.ts` + `index.ts` (the catalog/registry); `components/OpeningJourney/{OpeningJourney,ReadThrough,Drill}.tsx`; `app/openings/{page,[slug]/page}.tsx`. Decisions vs the plan: (1) **evals are 100% engine-sourced, never hand-typed** — `scripts/refresh-opening-evals.mjs` pulls White-POV cp from the Lichess **cloud-eval** API for every main-line FEN and bakes them into a committed `content/openings/evals.generated.json`, which `readSteps` reads (keyed by FEN, so transpositions share one eval); runtime stays static with zero network/LLM calls. (2) The **drill bot uses no engine** — it picks among the curated taught replies weighted by popularity, deviating only into lines the read-through taught; off-book → gentle correct-and-retry with a "show the move" reveal hatch. (3) `lib/openings/eco.ts`/`explorer.ts` were **not needed** — matching is against our own curated trees, not arbitrary openings (in-game opening awareness was split out to `ideas.md`). (4) `SiteHeader` became a client component so the nav highlights the active route. Verification: **33 new vitest tests** (81 total) covering every line's legality across all 4 openings, the read-through/FEN consistency, and the drill logic (in-book match tolerance, weighted bot selection, off-book rejection); browser-verified read-through + recall drill on both a White opening (Italian) and a Black one (Sicilian, where the bot opens automatically and the board flips), desktop + mobile, zero console errors.

### M6 — Play, reframed: Play + Openings tabs · a randomized board · hot-seat
Make **Play** a place you want to be. _Design locked via grilling — see decisions below._ **Build order: the randomized board UI first, then hot-seat multiplayer.** (The Coach tab — the third tab in the long-term vision — is its own later milestone, **M8**; M6 ships the two tabs whose content exists or is being built here.)

- **Tab restructure → Play · Openings (now); Coach (M8).** Nav drops the dead `Review`/`Progress` placeholders. M6 ships two live destinations — **Play** (the new work below) and **Openings** (the shipped M5 journeys). Coaching stays where it already works for now (the in-`/play` `CoachPanel` + move-list "explain", available in **bot mode only**); a standalone Coach tab is built in M8.
- **Play is one screen with a `Bot · Multiplayer` mode switch** over a *generalized* game loop. `useChessGame` is refactored from bot-coupled to opponent-aware (`bot | local`, with `remote` added in M7): the chess.js bookkeeping (sync / captures / end-detection / promotion / lastMove) is shared; only *who supplies the opponent move* and *takeback depth* differ. The randomized board is mode-agnostic, so it works in both.

- **Feature 1 — Randomized board (built first). A randomizer, not an editor.** Roll a **one-of-a-kind 2D board**; the classic **"Deep-Space Analysis Deck"** board is the default, and **reset-to-default** is always one tap away. A live **reroll** retints the *current* position instantly (no pre-game gate); the rolled board **sticks until changed**, persists across sessions (`localStorage`), and carries a short **seed** (the board's identity — reused by the M7 share-link so both players see the same board).
  - **Scope (Q-grill = B):** a cohesive theme = square palette + piece tones + board frame + a *derived* interactive accent. The four on-board cues (last-move / selection / check / capture) stay mutually distinguishable via **shape as well as hue**.
  - **Generation (Q-grill = C, hybrid — "constrain first, vary second"):** curated **families** + seed-jittered hue/chroma **within OKLCH clamps** + a hard contrast backstop. Engineering spine in `research/randomizer-color-system.md` (researched): library **`culori`** (MIT — guardrail-clean), seed via **`xmur3`+`mulberry32`**, **monochromatic board + near-neutral pieces at L extremes + split-complementary accent**, **piece-vs-square ≥ 3:1** (WCAG 1.4.11, against *both* squares), **square ΔL ≥ 0.28**, **mud-zone ban** (no L .40–.60 + C .05–.12), ~6 weighted families (warm wood / cool slate / ink & parchment / sage / marble / jewel). Every roll is ugly-by-construction-impossible.
  - **Accents (Q-grill = B):** the board's interactive cues re-hue to harmonize (contrast-guaranteed against the squares); **check stays red** (safety); **all side-panel analysis/classification colours stay fixed** (there the colour *is* the data).
  - **Pieces (Q-grill = A):** recolor + **CSS-only material treatment** (rim / sheen / emboss); unicode glyphs unchanged; **no alternate piece sets** (GPL minefield — parked in `ideas.md`).
  - **Delight:** **named boards** (`Family · seed`) + a **signature staggered re-tint sweep** on reroll; `prefers-reduced-motion` → instant swap.
  - **Design workflow (per `distinctive_design.md`):** explore *before* implementing — author standalone, browser-renderable mockups in `research/designs/` (the candidate families on a real board with pieces + deliberately-extreme rolls to pressure-test legibility, and **2–3 distinct directions for the reroll/naming experience**), render + screenshot-self-review, then **"port the decisions, not the code"** into the generator + Board theming layer. Principle-based direction, banned-defaults as counter-pressure, motion by trigger/element/feel, a11y baked in.
  - _Scoped out → `ideas.md`: a manual customizer/editor, alternate piece sets, a 3D board._

- **Feature 2 — Local hot-seat 2-player (built second).** Two humans, one device, via the `Multiplayer` mode. **No coach, no engine analysis** during a hot-seat game (a live best-line readout next to two competing humans is an engine assist — and Play is "just play"). **Static** orientation (White at the bottom) + the existing manual **Flip** (auto-flip parked). Full check/checkmate/stalemate/draw detection; **takeback undoes 1 ply** (not 2); **neutral** result text ("White is mated", not "the bot mated you"); the right panel hides the strength + "your side" selectors and shows **two player strips** (labelled White / Black; name entry is a later nicety); resign is per-side; **no clocks** in M6.

- **Done when:** the app presents **two working tabs** (Play / Openings); on Play you can switch `Bot ⇄ Multiplayer`; you can **roll a unique randomized 2D board or keep the classic default** (and every roll is legible + on-brand); two people can play a **complete legal hot-seat game** on one device with correct turn-taking and end-state detection; bot mode (with its coach + analysis) still works unchanged; everything meets the `DESIGN.md` bar on desktop + mobile with reduced-motion fallbacks.
- **Key files:** `research/designs/` (board-theme + reroll exploration mockups) and `research/randomizer-color-system.md` (the colour-system spec); `components/SiteHeader.tsx` (nav → Play / Openings); `lib/board/theme.ts` (families + seeded OKLCH generator + contrast backstop + seed serialize/deserialize), `lib/board/rng.ts` (`xmur3`/`mulberry32`); `components/BoardRandomizer/` (reroll / name / reset-to-default UI + the re-tint sweep); `components/Board/` (consume a theme); `components/Play/` (the `Bot ⇄ Multiplayer` mode switch + hot-seat path in `useChessGame.ts` + `PlayClient.tsx`).

- **✅ Shipped (M6).** Play is now two live tabs (**Play · Openings**; the dead Review/Progress placeholders are gone — `components/SiteHeader.tsx`) with a `Bot ⇄ Multiplayer` switch over one generalized loop. Built **exploration-first**: `research/designs/board-randomizer.html` runs the real generator on a real board (6-family gallery + a 48-roll legibility stress test + the three reroll/naming directions) — the review gate that locked the decisions before any prod code. Actual files (under `src/`): the **randomizer** is `lib/board/rng.ts` (`xmur3`/`mulberry32` + `seededRandom`/`shortId`) + `lib/board/theme.ts` (the 6 OKLCH families, the seeded generator, the contrast/ΔL/mud-zone backstop, `themeForSeed`/`randomBoardSeed`, and the classic `CLASSIC_THEME`), using **`culori`** (MIT, guardrail-clean) for gamut-clamping + CSS emission; the Board consumes a theme as inline `oklch()` CSS vars (`components/Board/Board.tsx`, with `globals.css` routed through `--board-*` vars that fall back to the classic deck); `components/BoardRandomizer/{BoardRandomizer,useBoardTheme}.tsx` is the **"Dealer's roll"** control (reroll / reset-to-classic / copy-seed-link, `localStorage` persistence, named boards `Family · seed`, the staggered re-tint sweep with a `prefers-reduced-motion` instant-swap). `components/Play/{PlayClient,useChessGame}.tsx` carry the opponent-aware loop. Decisions vs the plan: (1) **legibility is guaranteed on the piece, not the rim alone** — pieces are *two-tone* (fill + opposite-luminance outline) and the floor is `max(contrast(fill,sq),contrast(rim,sq)) ≥ 3` against *both* squares; the research doc's "near-white fill ≥ 3:1 vs the light square" is geometrically impossible for a bright wood board, so the rim/material treatment is what frees squares to be bright and varied (the headline finding from the exploration mockup). (2) the **mud-zone ban** is enforced on the accent only — the family table's own dark squares (e.g. warm-wood `L.50 C.065`) sit inside the banned band, so squares are governed by `C ≤ 0.08 ∧ ΔL ≥ 0.28` instead. (3) on-board cues **re-hue to the family accent** (distinguished by shape); **check stays red** and all side-panel classification colours stay fixed; the classic default keeps amber last-move. (4) reroll UX = **direction A "Dealer's roll"** (one verb) **+ the shareable seed** folded in (the M7 link reuses it verbatim). (5) hot-seat shares the chess.js bookkeeping; only *who supplies the opponent move* (engine vs none), *takeback depth* (2 vs 1 ply) and *result text* (neutral) differ, and Multiplayer surfaces **no coach/analysis**. Verification: **12 new vitest tests** (now **93** total) prove every roll passes both floors across ~660 seeds (incl. adversarial: low ints, empty, unicode, long), the accent never lands in the mud-zone, same seed → same board, and all six families are reachable; the generator was also swept headless over **20,000 rolls — zero failures**, worst piece-contrast 3.60:1, worst square ΔL 0.30. Browser-verified desktop + mobile (390px): rerolls re-tint legibly and on-brand, persist across reload, reset restores the classic deck; a hot-seat game plays both sides with correct turn-taking, single-ply takeback and neutral end text; bot mode (coach + analysis) is unchanged. `tsc`, `eslint`, and the full suite are clean.

### M6.5 — The Generative Table: a live-LLM-invented UI per game
The M6 randomizer re-tinted the *board*. This evolves it into a **per-game generative "table"**: clicking re-roll makes a **live Claude call** that invents a complete, fresh, coherent visual world for that one game — and the player plays on it. Play tab only.
- **What changes per game (Q-grill outcome):** surfaces & colour, typography, shape language, board palette, **and the pieces** — but **nothing functional** (game logic, the coach pipeline, and the coach voice are untouched). One reskinned layout (not a structural reflow).
- **LLM designs, we render (the safety line):** Claude returns a **validated design spec** (a tool-call JSON), never raw CSS. The renderer maps it onto the app's existing `:root` tokens (so the whole Play screen reskins with zero CSS rewrites) and a **deterministic legibility backstop** (sRGB WCAG, ported from the M6 generator) repairs any spec so the board hero can never come back illegible — light *or* dark worlds. Degrades to a built-in fallback world if the key/API is unavailable.
- **Genuine novelty, not presets (per `research/randomize-prompt-research.md`):** a tiny random brief collapses to "concept-magnet" AI defaults; the generator fights this with a **combinatorial high-entropy brief** (era × light × material × movement × feeling + a cognitive-dissonance clash) and a **forced value-key** so worlds come back paper-bright as often as dark.
- **Pieces:** recolorable real silhouette sets (canonical, recognizable — never LLM-drawn-from-scratch) + per-world treatments (filled / flat / bold / neon) + an authored letter-mark set; SVG fills/strokes are CSS-driven, so pieces recolor to any world. **Permissive licensing only** — chessnut (Apache-2.0) ships; maurimo (MIT) / RhosGFX (CC0) / kiwen-suwi (CC-BY) / cburnett-BSD are the drop-in family backlog.
- **Ephemeral:** a dealt table is **not persisted** — the classic Deep-Space look is always the default (drops M6's `localStorage` board persistence). It **subsumes** the board-randomizer control into one "Deal a table" / Reset.
- **Done when:** on Play, "Deal a table" runs a live generation (~15–30s, a designed loading moment) and the whole screen — board, pieces, panel, coach, controls, type — reskins into a coherent, legible world; classic is the default and a refresh returns to it; Openings/Coach are untouched; bot + hot-seat play unchanged.
- **Key files:** `lib/table/` (`spec.ts` types + token mapping, `backstop.ts`, `generate.ts` server Claude call + entropy engine + fallback, `pieceSets.ts`), `app/api/table/route.ts`, `components/Play/{useGameTable,TableDealer}.tsx`, `components/Board/Board.tsx` (SVG pieces), `globals.css`; explored in `research/designs/generative-table.html` + `table-server.mjs` + `build-pieces.mjs`.
- **✅ Shipped (M6.5).** Live generation, the entropy engine, the legibility backstop, the token mapping, and recolorable SVG pieces (chessnut + letter) are all in the app, Play-tab only and ephemeral. Built exploration-first (the standalone sandbox proved the range across light/dark worlds + a live `/api/generate-table` server before any prod wiring). **Deferred:** shape/motion **variants** (corner/frame/motion are captured in the spec + on the wrapper as data-attrs, but the variant CSS isn't built yet — every world currently keeps the Deep-Space HUD frame *recolored* rather than morphing); header/footer stay Deep-Space (scoped to `.game`); the **share link** (folds into M7); **more piece silhouette families** (the permissive backlog above). The old `BoardRandomizer`/`useBoardTheme`/`lib/board` are now unused.

### M7 — Play online: a shareable live link
The headline. Two people, two devices, one live game — share a link, play in real time. The one deliberate step beyond local-first (see Architecture).
- **Shareable room link.** "Play a friend" mints an unguessable room link (the **randomized board seed travels in the link**, so both sides see the same board). The opener joins as the other color; no accounts, ephemeral + private by default.
- **Realtime transport.** A small relay carries *only* live game state — moves (SAN/UCI), the seed, colour assignment, presence, resign/draw offers. **Approach decided at build (recommend a hosted realtime service for reliability over P2P+signaling);** must handle late-join, **reconnect-after-refresh**, and "opponent left." Transport only — not a database of record, never near the UCI boundary.
- **Trust model.** Each client is the authority on its *own* legal moves via `chess.js`, and **both clients independently re-validate** every incoming move (a friendly game needs no trusted server; a server check can come later). Illegal/desynced input is rejected, not rendered.
- **Done when:** Rory generates a link, sends it to a friend, and the two play a complete legal game live across two devices with correct turn-taking, shared randomized board, working resign/draw, correct end-state — and a mid-game refresh reconnects into the same position.
- **Key files:** `app/play/[room]/page.tsx` (room route), `lib/realtime/` (transport abstraction + chosen provider adapter), `lib/board/theme.ts` (seed ↔ link), `components/Play/` (online mode), optionally `app/api/room/` (room mint / signaling).

## Key risks (carry forward)
- **Coaching/opening quality is everything** — mitigate with strict grounding + move validation + deterministic classification (the M2 spine is the defense).
- **LLM hallucinating chess** — never let Claude compute chess; validate every named move.
- **GPL contamination of our code** — permissive libs only; isolate Stockfish behind the UCI boundary; no `chessground`/`chessops`.
- **COOP/COEP misconfig on Vercel** — avoided by starting single-threaded.
- **WASM size / mobile-Safari perf** — lite build, CDN-served static asset, depth throttling, fallback build.
- **LLM cost creep** — on-demand calls, compact payloads, per-position caching, streaming.
- **Lichess API rate limits** — opening explorer used as a cache, one request at a time, back off on 429.
- **Randomized board hurting legibility/taste (M6)** — constrain the generator to vetted palette/contrast ranges so *every* roll passes `DESIGN.md` + WCAG; classic-board opt-out is always available; the randomizer never overrides reduced-motion.
- **Realtime breaking local-first (M7)** — keep the relay a transport for live game state only (moves + seed + presence), ephemeral and account-free; never the source of truth and never near the UCI boundary; both clients re-validate every move. Adds infra cost + a drop/reconnect failure mode (handle explicitly, degrade gracefully).
